## 1. Executive Summary

Habitiq is a **shared living management platform** that solves one of the most common and persistent frictions in co-living: who does what, when, and whether it actually gets done.

Every flat, hostel room, PG accommodation, and co-living space faces the same recurring issue — household duties are assigned informally, tracked poorly, and enforced through awkward conversations. Habitiq replaces this entire process with a transparent, automated, real-time rotation system that all flatmates can see and trust.

Built for the Indian urban market — where tens of millions of young professionals and students live in shared accommodations — Habitiq is a mobile-first web application requiring no installation, no account setup friction, and no coordination overhead. A flat admin creates a flat in under 60 seconds, shares an invite code, and the system begins managing duty rotation automatically from day one.

Habitiq is currently live with real users in the trial phase. The product has been security-audited, is hosted on enterprise-grade infrastructure, and is architected to scale from 10 flats to 10,000 with minimal infrastructure cost.

**The opportunity is large.** India alone has an estimated 10+ million people living in shared accommodations in Tier 1 and Tier 2 cities. No dominant product exists in this space. Habitiq is positioned to become the default tool for shared living management in India — and eventually globally.

## 3. The Problem

### The Shared Living Reality

When two or more people share a living space, household duties become a social contract. Someone has to clean the bathroom. Someone has to take out the trash. Someone has to mop the floor. In the absence of a system, this contract is informal, verbal, and fragile.

The result is predictable:

- **Accountability gaps.** "I forgot" or "I thought it was your turn" become recurring excuses.
- **Unfair distribution.** The most conscientious person ends up doing most of the work. The least conscientious person free-rides.
- **Social friction.** Asking someone to do their chore feels like a confrontation. Nagging creates resentment. Silence creates mess.
- **No visibility.** Nobody knows who did what last week, who has the worst completion rate, or who has been consistently reliable.
- **Travel disruption.** When one person travels, their duties get skipped, dumped on others, or forgotten entirely.
- **Swap chaos.** Negotiating task swaps happens over WhatsApp in group chats that everyone has muted.

### The Status Quo

Most flats today manage duties in one of three ways — all of them broken:

| Method | Why It Fails |
|--------|-------------|
| WhatsApp group | Messages get buried, no accountability, no history |
| Whiteboard / sticky notes | Not visible when you're out, gets erased, no audit trail |
| Verbal agreement | No record, relies on everyone's memory, breaks down immediately |
| Excel / Google Sheets | Requires maintenance, no notifications, feels like work |

### The Gap in the Market

Productivity apps like Notion or Todoist are designed for individuals or work teams. They are far too complex for a flat of 4 students who just want to know whose turn it is to clean the kitchen. Expense tracking apps like Splitwise don't touch duties at all. There is no purpose-built, simple, shared living operations tool in the Indian market.

**Habitiq fills that gap.**

## 4. The Solution

Habitiq replaces every informal, broken system for managing household duties with a single, transparent platform that:

1. **Automatically rotates** duties among all flat members on a defined schedule
2. **Skips absent members** intelligently and resumes their rotation when they return
3. **Tracks completion** in real time so everyone can see what is done and what is overdue
4. **Enables task swaps** through a formal request-and-accept flow instead of WhatsApp chaos
5. **Gives every member analytics** — who is pulling their weight, who is falling behind
6. **Requires zero coordination** from the admin once set up — the system runs itself

### Core Insight

Habitiq is not a to-do app. It is a **fairness engine**. The core value proposition is not just task tracking — it is the elimination of the social friction that comes from perceived unfairness. When the system assigns duties transparently and everyone can see the rotation, there are no arguments. The app becomes the authority, not a person.

### Smart Rotation Engine

This is the core of Habitiq — the algorithm that decides who does what and when.

| Feature | Description |
|---------|-------------|
| Auto-assignment | When a task becomes due, it is automatically assigned to the next person in the rotation queue |

| Manual override | Admin can assign any task to any member at any time, overriding the rotation |
| Custom start date | When creating a task, admin can set a custom calendar start date for the first due date |
| Task frequency | Tasks can be set to daily, weekly, fortnightly, or monthly frequency |

    ↓
Scrolls to "Danger Zone"
    ↓

    ↓

## 9. Technical Foundation

Habitiq is built on a modern, production-grade technology stack designed for reliability, speed, and scale.

### Technology Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Frontend Framework | Next.js 16 (React 19) | Server rendering, routing, performance |
| Build Tool | Turbopack | Sub-second hot reload, fast production builds |
| Styling | Tailwind CSS v4 | Utility-first, consistent design system |
| State Management | Zustand | Lightweight, predictable, no boilerplate |
| Icons | Lucide React | Clean, consistent icon set |
| Backend / Database | Firebase Firestore | Real-time sync, scalable NoSQL, offline support |
| Authentication | Firebase Auth | Google + email/password, secure, scalable |
| Hosting | Vercel (Hobby) | Native Next.js support, 6,000 build mins/month |
| Auth Proxy | Next.js rewrites | Custom authDomain for mobile login reliability |

### Architecture Highlights

**Real-time by default.** Firestore's `onSnapshot` listeners mean every member sees changes the moment they happen. No polling, no page refresh, no stale data.

**Rotation engine is deterministic.** The rotation algorithm is pure logic: given a task's queue and the list of active members, the next assignee is always predictable and consistent — regardless of which device runs the calculation.

**Offline-ready foundation.** Firestore's local persistence layer means the app can serve cached data when connectivity drops, with full sync on reconnect. (Full offline mode is a Phase 3 feature.)

**Mock mode for development.** The entire app runs without Firebase keys using seeded mock data — enabling rapid development and testing without touching the production database.

**Security is structural.** Firestore Security Rules enforce role-based access at the database level — not just in the UI. Even if a client is compromised, the rules prevent unauthorised reads and writes.

## 10. Security & Trust

A full security audit was completed prior to launch. Habitiq follows security best practices at every layer.

### Security Measures Implemented

| Area | Measure |
|------|---------|
| Firestore Rules | Role-based access control — members can only read/write their own flat's data |
| Task operations | Only admin can create or delete tasks |
| Swap requests | Only the receiving member can accept/decline their own swap request |
| Activity log | User ID on activity entries is server-validated, not client-supplied |
| HTTP headers | 6 security headers configured (CSP, HSTS, X-Frame-Options, etc.) |
| Auth error messages | Sanitised — no user enumeration possible |
| Input validation | All inputs have maxLength limits to prevent injection |
| ID generation | `crypto.randomUUID()` — cryptographically secure, not `Math.random()` |
| External assets | All third-party assets localised — no unvetted CDN calls |
| Firebase API keys | Client-side by design (Firebase standard) — protected entirely by Firestore Rules |
| Google OAuth | Custom auth domain proxy ensures mobile login works securely across all browsers |
| Data scope | Only email and display name collected from Google — no calendars, contacts, or drive access |

### Data Privacy

- No user data is sold or shared with third parties
- No advertising trackers
- Only the minimum data required to operate the product is stored
- Users can leave or delete their flat at any time, removing their data from the system

## 16. How Habitiq Was Built

### A New Kind of Founding Story

Habitiq was not built the traditional way. There was no large engineering team, no months of sprint planning, no agency. The entire product — from the first screen to a live, security-audited, real-time application used by real flatmates — was built by one person with a clear product vision and a new way of working: **AI-assisted vibe coding**.

This is worth documenting not just as a historical note, but because it is a core part of what Habitiq is and how fast it can move.

---

### The Builder: Venkata Sai Jaswanth E

Jaswanth is a UI/UX designer and product thinker — not a traditional software engineer. He came to Habitiq with deep knowledge of how people actually behave in shared living situations, a strong visual instinct, and a clear sense of what the product needed to feel like. ### The Method: Vibe Coding with AI

**Vibe coding** is the practice of building software by describing what you want in plain language and working collaboratively with an AI coding assistant to produce the actual code. The human brings the vision, the judgement, the product sense, and the taste. The AI brings the implementation speed, the framework knowledge, and the technical execution.

Jaswanth used this method end-to-end to build Habitiq:

- **What he brought:** the product concept, the user flows, the design decisions, the feature prioritisation, the business logic, the brand identity
- **What Claude Code brought:** the full implementation — Next.js components, Firestore queries, Zustand store design, auth flows, security rules, Tailwind styling, deployment configuration

Every screen, every feature, every line of Firestore security rules was produced through this collaborative loop: describe the intent → build it → test it → refine it → ship it.

---

### The AI Stack

Habitiq was built using two AI tools working in complementary roles:

#### Claude Code — Primary Builder

Claude Code was the primary development partner throughout the entire build. It handled:

- Writing all React components and Next.js pages from scratch
- Designing and implementing the Firestore data schema
- Building the rotation engine algorithm

Claude Code did not just generate boilerplate — it reasoned through architectural decisions, caught security vulnerabilities before they shipped, and iterated rapidly on feedback.

#### Gemini — Sounding Board & Knowledge Check

Google Gemini was used as a secondary tool for a different purpose: **validation and knowledge expansion**.

When Jaswanth was forming the product concept, use cases, and business logic, he used Gemini to:

- Fill in backend architecture knowledge (how Firestore transactions work, what patterns to use for member counts, etc.)

Gemini acted as the thinking partner. Claude Code acted as the building partner. The product vision came entirely from Jaswanth.

## Appendix A — Product Principles

These are the principles that guide every feature decision at Habitiq:

1. **Fairness is the product.** Every feature exists to make shared living more fair and less argumentative.
2. **Zero friction onboarding.** A flat must be up and running in under 2 minutes. If setup is painful, the product fails before it starts.
3. **Transparency by default.** All members can see everything. No hidden information. The visibility itself is the accountability mechanism.
4. **The system is the authority.** The product must take the burden of enforcement off individual people. Nobody should have to nag. The app does it.
5. **Mobile first, always.** The majority of users will open Habitiq on their phone while standing in the kitchen. Every decision optimises for that moment.
6. **Real-time or nothing.** If changes are not reflected instantly, trust in the system breaks down. Real-time sync is non-negotiable.
